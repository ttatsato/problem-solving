# Clone Graph

## どんな問題か
「ポインタが絡み合った構造を、参照関係を維持したままコピーせよ」という問題。  
グラフに限らず、循環参照を含む任意のデータ構造のディープコピーを正しく実装できるか、が問われている。

## 本質的な問い
「すでに処理したものを覚えているか」

これだけ。
循環・重複・再訪問、すべての問題は visited マップ1つで解決する。DFSかBFSかは手段に過ぎない。
visited に入っている → そのコピーを返す
visited に入っていない → コピーを作ってから visited に登録して進む
この2行の判断を正確に実装できれば解ける。

## Clone Graph のユースケース
「グラフをディープコピーする」という操作が実際に必要になる場面。

## 1. ゲームの状態管理
探索アルゴリズム（AIの先読みなど）で「現在の盤面状態を壊さずに試行錯誤したい」とき。
現在の状態グラフをclone
→ cloneに対して「この手を打ったらどうなるか」をシミュレート
→ 元の状態は無傷のまま

チェスのエンジンやパズルゲームのソルバーで使われるパターン。

## 2. Undo/Redo 機能
グラフ構造を持つアプリ（マインドマップ、フローチャートエディタ等）で操作前のスナップショットを保存する。
操作前にグラフをclone → historyスタックに積む
→ Undoしたいときにhistoryから復元

## 3. 並列処理・マルチスレッド
同じグラフを複数のスレッドで独立して処理したいとき、共有すると競合が起きるのでコピーを渡す。


現実での頻度
正直に言うと、Clone Graph という問題を直接解くシーンは少ない。
実務では：

グラフ構造には既存ライブラリが深いコピーメソッドを持っていることが多い
そもそもイミュータブルなデータ構造で設計して「コピーが不要な設計」にすることも多い
